# Phase 9: Shiki Syntax Highlighting Implementation Plan

This document outlines the architectural approach for integrating [Shiki](https://shiki.style/) to provide beautiful, accurate, and performant syntax highlighting for all Markdown code blocks (` ``` `) across the Enterprise Documentation Platform.

## Motivation
Currently, code blocks are rendered as raw `<pre><code>` HTML elements by `marked.js` without any tokenization or coloring. By integrating Shiki, we will parse the code during the Static Site Generation (SSG) and Just-in-Time (JIT) compilation phases on the backend. Shiki uses TextMate grammars (the exact same syntax rules as VS Code), ensuring perfectly accurate coloring across dozens of languages without injecting heavy client-side JavaScript bundles into the Angular application.

## Proposed Changes

---

### Component 1: Renderer Library (`libs/renderer`)
This is the core markdown compilation engine running on the NestJS Backend and CLI.

#### [MODIFY] `libs/renderer/package.json`
- Install `shiki` as a production dependency: `npm install shiki`
- Note: Shiki is ESM-native so we must ensure Node ecosystem compatibility.

#### [NEW] `libs/renderer/src/lib/plugins/shiki.plugin.ts`
Implement a singleton wrapper and a Marked custom extension.
1. **Singleton Initialization**: Create an async factory `getShikiHighlighter()` that invokes `createHighlighter` from `shiki`. This ensures the heavy WASM grammars are only loaded into backend memory once per application lifecycle.
2. **Themes**: Configure the highlighter to load a dual-theme setup corresponding to the Portal's UI toggle (e.g., `github-light` and `github-dark`).
3. **Marked Extension**: We cannot simply use the synchronous `marked.use()` for Shiki because WASM grammar loading and parsing (`highlighter.codeToHtml()`) are inherently asynchronous. 
   - We must leverage the `marked.parse(..., { async: true })` pipeline.
   - We will write a custom `marked` renderer specifically targeting the `code(code, infostring)` token. 
   - When marked encounters a code block, this override will await the Shiki WASM engine, passing the raw string, the parsed language (`infostring` like `typescript` or `yaml`), and output raw, tokenized, and inline-styled HTML spans.

#### [MODIFY] `libs/renderer/src/lib/renderer.ts`
Update the primary exported `renderMarkdown(rawMd: string)` pipeline:
1. Ensure the enclosing function is strictly `async`.
2. Await the Shiki Highlighter initialization.
3. Inject the custom Shiki rendering logic into `marked`.
4. Add the generated `<pre class="shiki">` and nested `<span style="...">` tags to the `DOMPurify` whitelist so the sanitizer does not immediately strip the beautiful inline coloring injected by Shiki.

---

### Component 2: Frontend Design System (`libs/theme` / `apps/portal`)
Even though Shiki injects explicit inline CSS properties (`<span style="color: #FF0000">`), the parent `<pre>` block still needs structural CSS and theme mapping.

#### [MODIFY] `apps/portal/src/styles.css` (or equivalent theme file)
1. **Container Styling**: Add smooth padding, border-radius, horizontal scrolling (`overflow-x: auto`), and a distinct background color to `.shiki` containers to make them visually pop off the page.
2. **Theme Toggling**: Shiki supports CSS variable mapping for dynamic light/dark mode. We will extract the inline styles generated by Shiki and map them to standard `--shiki-light-bg` and `--shiki-dark-bg` variables triggered by the `body.dark-theme` class already present in the Angular application.

---

### Component 3: Build Validations (`apps/api` / `libs/cli`)
Since Shiki fundamentally shifts the core markdown engine (`renderMarkdown`) to be an `async/await` pipeline, any consumer upstream must be verified.

#### [MODIFY] `libs/cli/src/lib/cli.ts`
- Ensure `buildStaticDocs` successfully awaits the `renderMarkdown` payload before stringifying it to disk.

#### [MODIFY] `libs/backend/src/lib/storage/local-disk.service.ts`
- Ensure `getDocumentContent` correctly awaits `renderMarkdown(rawMd)` when serving non-prebuilt JIT content dynamically to the Angular frontend.

## Verification Plan

### Automated Tests
1. **Unit Tests (`libs/renderer`)**:
   - Create `shiki.plugin.spec.ts`.
   - Mock a raw markdown string containing a ```typescript` block.
   - Assert that the output HTML successfully contains `<pre class="shiki">` and nested tokenized `<span>` tags.

2. **Playwright E2E Validation (`apps/portal-e2e`)**:
   - Enhance the portal test suite to navigate to a sample document containing a code block.
   - Add strict assertions to locate the `.shiki` injected DOM nodes within the browser.
   - Verify that nested `<span>` elements contain inline `color` CSS style variables explicitly guaranteeing the Shiki text-mate grammar engine successfully executed over the network.

### Manual Verification
1. Open the `/sample-docs` repository and add a complex TypeScript or YAML code block to an `overview.md` file.
2. Run `npx nx serve api` and `npx nx serve portal`.
3. Navigate to that document in the browser.
4. Verify the code block is beautifully syntax-highlighted matching the active VS Code theme.
5. Toggle the Dark/Light mode button in the Portal Header and verify the code block background and token colors seamlessly invert locally via CSS.
